## プロジェクト提案書：GNNを用いた地方大学生向け居住地推薦システム## 1. 背景と社会的意義（地域課題の解決）

* 地方都市の移動弱者問題：函館をはじめとする地方都市は高度な車社会であり、自家用車を持たない新入生や転居者は、交通機関の格差や冬期の移動リスク（路面凍結・積雪）に直面しやすい。
* 情報の非対称性：インターネット上の物件情報だけでは、店舗の網羅性や「リアルなバスの利便性（深夜運行の有無など）」を把握できず、入居後の生活やアルバイト選択でミスマッチが起きている。
* 地域公共交通への貢献：学生が公共交通（路線バス）を利用しやすい最適な地域を可視化・推薦することで、地域の足（函館バスなど）の維持・利用促進という地域課題の解決に寄与する。

------------------------------
## 2. システム概要
ユーザー（新入生等）が「所属大学」を選択し、「重視するライフスタイル（通学、買い物、バイト・移動）」の優先度を直感的に設定すると、裏側のGNNモデルが地域のネットワーク構造を解析し、最適なおすすめ居住エリアをランキング形式で提案するシステム。
------------------------------
## 3. GNN（グラフニューラルネットワーク）を用いたアプローチ## ① なぜGNNか？
従来の「スーパーから〇〇m以内なら10点」といった静的な足し算（ルールベース）ではなく、「住まい」「店舗」「バスの結びつき」を一つのネットワーク（異種グラフ構造）として捉え、構造そのものを学習させることで、よりロジカルで多角的な利便性評価を実現する。
## ② グラフ構造の定義

* ノード（点）：
* 居住エリアノード（例：美原1丁目、五稜郭町など）
   * 店舗ノード（スーパー、コンビニ、ドラッグストアなど）
   * 目的地ノード（各大学、主要な商業・バイト密集地、函館駅など）
* エッジ（線）と特徴量：
* 居住エリア ⇄ 店舗：物理的距離（徒歩圏内かどうかの判定に使用）
   * 居住エリア ⇄ 目的地：路線バスの運行データ（通学・バイトの移動しやすさを表現）

------------------------------
## 4. データベース設計（スプレッドシート項目）
データ収集の現実的な割り切りと規約（Google規約の回避等）を考慮し、「公式オープンデータ」や主要スポットに限定したサンプルデータを蓄積・活用する。
## シート1：店舗情報（ノード用）

| カラム名 | データ型 | 説明 | 役割 |
|---|---|---|---|
| shop_id | String | 店舗のユニークID（例: S001） | 主キー（エッジ用） |
| shop_name | String | 店舗名 | 画面表示用 |
| category | String | supermarket, convenience, restaurantなど | ノードの分類 |
| latitude / longitude | Float | 緯度・経度 | 居住エリアとの距離計算用 |
| is_chain | Boolean | チェーン店フラグ（1: はい / 0: いいえ） | バイトの雇用の多さの指標 |
| opening_hours | String | 営業時間（例: 24hours, until_22） | 夜間利便性の指標 |

## シート2：路線バス運行情報（エッジ用）
※函館バスのGTFS（標準的なバス情報フォーマット）オープンデータから必要路線を抽出。

| カラム名 | データ型 | 説明 | 役割 |
|---|---|---|---|
| route_id | String | ルートのユニークID | 主キー |
| origin_area | String | 出発地（居住候補エリア） | ノードAの特定 |
| destination_id | String | 目的地ID（大学や五稜郭などのエリア） | ノードBの特定 |
| daily_trips | Integer | 1日の運行総本数 | 移動の基本利便性（エッジの重み） |
| night_trips | Integer | 21時以降の運行本数 | バイト帰りの安心度（エッジの重み） |
| duration_min | Integer | 目的地までの所要時間（分） | 通学・通勤の負担度 |
